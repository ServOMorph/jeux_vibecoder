# Session de brainstorming — Jeu pour apprendre à vibecoder
**Date :** 2026-07-17
**Référence :** *The Farmer Was Replaced* (apprendre Python via un drone agricole)

---

## Phase 0 — Cadrage

### Public cible
- Développeurs qui découvrent les agents de codage IA (Claude Code, Cursor, Copilot…) — pas des débutants absolus en code, débutants en *vibecoding*.

### Compétences de vibecoding à enseigner
- Formuler de bons prompts
- Décomposer un problème
- Itérer & corriger (lire les sorties de l'IA, reformuler, déboguer avec l'IA)
- Valider & garder le contrôle (relire le code généré, ne pas faire confiance aveuglément)

### Contraintes / concept clé du jeu
- **Le jeu n'a pas d'interface autonome : le jeu, c'est le code lui-même.**
- Le joueur reçoit un vrai projet (fichiers Python, avec un point d'entrée `run.py`).
- Il ouvre son **agent de codage IA réel** (Claude Code, Cursor, etc.) dans son environnement habituel.
- Il "joue" en écrivant des **prompts dans le chat de l'agent**, qui modifie les fichiers du projet.
- Il lance `run.py` pour observer le résultat concret (le jeu tourne, plante, ou fait autre chose selon la qualité du prompt).
- La progression = compétence à *diriger un agent*, pas à écrire du code à la main.
- → Pas d'IA simulée dans un jeu web : un vrai agent, sur un vrai repo, avec des défis/niveaux structurés comme un jeu.
- Plateforme : projet local (Python), pas un jeu web autonome.

### Questions "Comment pourrions-nous…" (HMW) — validées
1. Comment pourrions-nous faire ressentir à un développeur la différence entre "écrire du code" et "diriger un agent qui écrit du code" ?
2. Comment pourrions-nous rendre visible, et donc jouable, la qualité d'un prompt (mauvais prompt → mauvais résultat, bon prompt → résultat propre) ?
3. Comment pourrions-nous transformer le réflexe de décomposition de problème en mécanique de jeu gratifiante (comme TFWR transforme l'automatisation en gratification) ?
4. Comment pourrions-nous concevoir des défis de code (niveaux, objectifs, contraintes) qui obligent le joueur à bien prompter son agent, plutôt que de simuler nous-mêmes une IA ?
5. Comment pourrions-nous apprendre au joueur à *ne pas* faire confiance aveuglément à l'IA — repérer les erreurs, hallucinations, code cassé — sans que ce soit punitif ou décourageant ?

## Phase 1 — Divergence
_(idées au fil de l'eau)_

### 1a. Analyse du modèle (The Farmer Was Replaced)

| Mécanique TFWR | Équivalent vibecoding |
|---|---|
| Boucle coder → observer → optimiser | Boucle **prompter → observer → reformuler** (lancer `run.py` pour vérifier l'intention) |
| Déblocage de concepts un par un (variables → boucles → fonctions…) | Déblocage de **compétences de prompt** : prompts simples → contexte+contraintes → décomposition → auto-validation par l'agent |
| Tâche répétitive rendue gratifiante par l'automatisation | Tâche fastidieuse à la main rendue rapide par un bon prompt décomposé (l'"aha" du vibecoding) |
| Feedback visuel immédiat (drone, cultures) | `run.py` montre immédiatement si l'agent a compris : ça tourne / plante / niveau débloqué |
| Économie de progression (or → terrains) | Score/monnaie de "clarté" : bon prompt = moins d'itérations = progression rapide |
| Bac à sable limité qui s'élargit | Agent bridé au départ (contexte/fichiers limités), s'élargit avec la maîtrise |

### 1b. Crazy 8s — 8 concepts

1. **🏰 Le Donjon Inachevé** — roguelike aux étages cassés/incomplets ; prompter pour réparer l'étage avant de pouvoir y jouer. Boucle : jouer → morceau manquant → prompter → rejouer.
2. **🤖 L'Usine à Robots** — usine visuelle (pygame), chaque machine = module Python vide à faire implémenter. Boucle : goulot → décomposer → prompter → l'usine tourne.
3. **🕵️ Code Détective** — jeu "déjà vibecodé" plein de bugs subtils et hallucinations volontaires ; auditer/tester/corriger via l'agent. Compétence reine : ne pas faire confiance aveuglément.
4. **🚀 Mission Contrôle** — sonde/rover : le module de pilotage s'exécute en autonomie totale, aucune correction en vol. Compétence reine : spécifier et anticiper. Boucle : brief → spec précise → lancement → post-mortem.
5. **🧟 Tour de Garde** — tower defense temps réel, les tours sont des fonctions écrites par l'agent pendant que le jeu tourne (hot-reload). Prompt sous pression.
6. **🏗️ La Startup** — simulation CTO : tickets clients, satisfaction vs dette technique. Boucle : trier → décomposer en tâches promptables → livrer → conséquences.
7. **🎓 Le Golem** — élever une créature dont le comportement EST le code du projet ; chaque prompt modifie son "cerveau". Boucle : observer → diagnostiquer → ajuster → relâcher.
8. **⚔️ Arène de Prompts** — katas avec tests cachés et budget de prompts/tokens limité ; score au nombre de prompts. Leaderboard possible.

**Réactions utilisateur :**
1. Donjon Inachevé : ❤️ j'aime
2. Usine à Robots : ❤️❤️ **j'adore**
3. Code Détective : 🤔 bof
4. Mission Contrôle : ❤️ j'aime
5. Tour de Garde : ❤️❤️ **j'adore**
6. La Startup : ❤️ j'aime

→ Favoris nets : **Usine à Robots (2)** et **Tour de Garde (5)**. Écarté : Arène de Prompts (8). Tiède : Code Détective (3).

### 1c. SCAMPER — hybride « Usine + Tour de Garde »

Base : usine/base dont chaque module est du code Python écrit par l'agent ; phases calmes de construction + phases de défense sous pression.

- **S (Substituer l'univers)** : colonie spatiale (oxygène/énergie/serres, vagues = tempêtes/météorites) ; cité médiévale (forges, murailles vs hordes) ; écosystème/vaisseau vivant. Critère : justifier "modules indépendants + crises".
- **C (Combiner)** :
  - +Mission Contrôle : crises préparées à l'avance, module autonome, on regarde sans intervenir (spec/anticipation).
  - +Donjon Inachevé : démarrer avec une base en ruine au code cassé (lire du code avant d'en produire).
  - +Startup : la dette technique "tient" en phase calme mais **casse sous pression** pendant les vagues (validation = survie).
- **A (Adapter)** : incrémental/idle (production pendant qu'on prompte) ; roguelike (run = une nuit de siège, blueprints conservés) ; They Are Billions (longue préparation → vague massive annoncée, timer visible).
- **M (Amplifier)** : **hot-reload permanent** — chaque sauvegarde de l'agent rechargée en direct, l'usine évolue sous vos yeux. Signature du jeu.
- **P (Autres usages)** : outil de formation/onboarding en entreprise ; niveaux = fichiers de scénario créables par la communauté.
- **É (Éliminer)** : MVP = une machine + une vague. `defense.py` vide, vague dans 10 min, débrouillez-vous.
- **R (Renverser)** : le code des vagues ennemies est **visible et lisible** — demander à l'agent d'auditer le code ennemi pour concevoir la contre-mesure (récupère le bon côté de Code Détective).

**Réactions utilisateur sur le SCAMPER :** pas de tri à chaud — toutes les pistes partent au vote de la Phase 2.

### 1d. Brainstorming inversé — anti-patterns → principes de design

| Anti-pattern (échec garanti) | Principe de design inversé |
|---|---|
| Afficher le "prompt parfait" à recopier | Jamais de prompt tout fait : des objectifs et contraintes, le joueur formule seul |
| Aucun feedback après un prompt | Conséquence visible en <30 s via `run.py` |
| "Fais le niveau 3" marche aussi bien qu'un prompt soigné | Niveaux conçus pour que le prompt paresseux échoue de façon intéressante (ambiguïtés piégeuses, contraintes cachées, critères précis) |
| Murs de texte avant de jouer | Brief ≤ 3 lignes, le reste se découvre dans le code et par l'échec |
| Progression plate | Chaque niveau introduit une compétence nouvelle et nommée |
| Punir l'échec (tout recommencer) | Échec gratuit et instructif : on voit pourquoi, on reprompte |
| Tests opaques ("FAIL") | Critères de victoire lisibles — les lire via l'agent est une compétence enseignée |
| Dépendre d'un agent précis | Agnostique : fichiers Python + `run.py`, tout agent fonctionne |
| Tricher (éditer save.json) plus simple que jouer | Triche possible mais jeu plus intéressant qu'elle ; tricher élégamment = succès caché |

## Phase 2 — Convergence

### Mind map par thèmes

- **Concept de base (consensus fort)** : hybride Usine à Robots + Tour de Garde ; phases calmes (construction/modules) + phases de tension (vagues/défense) ; signature = hot-reload permanent.
- **Univers possibles** : colonie spatiale / cité médiévale / vaisseau-écosystème vivant.
- **Mécaniques empruntées** : modules autonomes sans correction en vol (Mission Contrôle) ; base en ruine à réparer d'abord (Donjon Inachevé) ; dette technique qui casse sous pression (Startup) ; code ennemi lisible à auditer (renversement de Code Détective).
- **Progression pédagogique** : compétences débloquées une à une ; score/monnaie de "clarté" ; agent bridé puis élargi.
- **Principes de design (garde-fous)** : pas de prompt tout fait, feedback <30s, échec instructif et gratuit, brief court, une compétence nommée par niveau, critères de victoire lisibles, agnostique à l'agent, triche possible mais peu intéressante.

### Vote par points (5 pts)
- Univers spatial : 2 pts
- Univers vaisseau/écosystème vivant : 3 pts
- → Fusion retenue : **vaisseau-écosystème dérivant dans l'espace**
- Aucun point sur les mécaniques → concept de base (Usine+Tour de Garde, hot-reload) déjà consensuel

### Grille d'évaluation — 3 variantes du concept

| Variante | Description | Pédagogie | Fun | Faisabilité |
|---|---|---|---|---|
| A — Minimale | Usine+Tour de Garde, hot-reload, vaisseau-écosystème, rien d'autre | 3/5 | 3/5 | 5/5 |
| **B — Ruine & Dette (RETENUE)** | + vaisseau en ruine au démarrage (lecture de code d'abord) + dette technique qui casse sous les vagues | 5/5 | 4/5 | 3/5 |
| C — Mission & Audit | + modules autonomes sans correction en vol + code ennemi lisible à auditer | 4/5 | 5/5 | 2/5 |

### Concept principal retenu
**Variante B — Ruine & Dette**, univers vaisseau-écosystème vivant dérivant dans l'espace.

## Phase 3 — Approfondissement

**Pitch** : Vous êtes seul aux commandes d'un vaisseau-écosystème à la dérive dont les systèmes sont un chantier de code inachevé ; en dirigeant votre agent IA de codage à coups de prompts précis, vous réparez, construisez et défendez votre arche vivante contre des vagues qui mettent votre code à l'épreuve.

**Boucle de gameplay** :
1. Observer l'état du vaisseau (dashboard texte/pygame minimal) : modules cassés, ressources, alerte de vague
2. Prioriser (réparer vs défendre vs optimiser)
3. Prompt ciblé dans le chat de l'agent, sur un fichier précis
4. L'agent modifie le code
5. `run.py` (hot-reload) → observer le résultat immédiat
6. Reprompter si besoin
7. Vague (timer visible) → les modules sont testés en live, dette technique craque sous pression
8. Bilan post-vague, compétence débloquée, cycle suivant

**Progression pédagogique** :
1. Premier signal (prompts simples, un seul fichier)
2. Contexte (contraintes précises, dépendances entre fichiers)
3. Découpage (objectif flou → sous-tâches promptables)
4. Lecture de code (comprendre du legacy avant de modifier)
5. Validation (demander tests/vérifications à l'agent)
6. Itération sous pression (reprompt rapide en vague)
7. Dette technique (arbitrer vitesse vs qualité)
8.+ Réservé extension "Mission & Audit"

**Rôle de l'IA** : le vrai agent de codage du joueur (Claude Code, Cursor, Copilot…) dans son propre environnement. Le jeu est 100% déterministe/scripté, aucun LLM intégré, agnostique à l'agent.

**Univers/thème/ton** : sci-fi contemplative, arche générationnelle + biosphère vivante. Ton entre solitude et espoir.

**Conditions de victoire/feedback/score** : feedback via `run.py` + dashboard (santé du vaisseau, jauge de dette technique) ; score de "clarté" (nb de prompts/itérations, non punitif) ; victoire de niveau = survivre à la vague ; victoire finale à affiner (stabiliser l'écosystème / atteindre une destination).

**MVP** : un module à réparer + une vague unique, dette technique visible, hot-reload fonctionnel, agnostique à l'agent, niveaux 1-4.

**Hors périmètre** : vraie API LLM interne, multijoueur, leaderboard compétitif, extension Mission & Audit, interface graphique riche.

**Risques/inconnues à prototyper** : faisabilité du hot-reload en direct ; fun sur la durée sans input direct dans le jeu ; équilibrage dette technique ; agnosticisme réel entre agents ; concevoir l'échec "intéressant" du prompt paresseux sans frustrer.
